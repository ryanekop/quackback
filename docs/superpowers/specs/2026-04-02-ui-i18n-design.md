# UI Internationalization (i18n) Design

**Date:** 2026-04-02
**Scope:** UI chrome i18n for widget and public portal (admin dashboard excluded)

## Problem

The widget and public portal are English-only, limiting adoption for companies with a global user base. Adding UI-level i18n keeps Quackback competitive for international teams and enables community-contributed translations.

This spec covers UI string translation, locale detection, and RTL layout support. Content i18n (translating admin-authored data like board names, status labels, help articles) is a separate follow-up.

## Dependencies & Tooling

### New Packages

- `react-intl` -- React bindings for FormatJS (`IntlProvider`, `FormattedMessage`, `useIntl`)
- `@formatjs/cli` (dev) -- string extraction and compilation

### CLI Scripts

Add to `apps/web/package.json`:

- `intl:extract` -- scans source for `<FormattedMessage>` / `defineMessages` and produces `en.json`
- `intl:compile` -- compiles locale JSONs into optimized AST format for production

No new Vite plugins needed. Vite handles JSON imports natively.

## Locale File Structure

```
apps/web/src/locales/
  en.json          # Source of truth (hand-authored + extraction-verified)
  fr.json          # Community-contributed
  de.json
  ar.json          # RTL locale
  ...
```

### Message ID Convention

Dot-separated, hierarchical: `{surface}.{component}.{category}.{descriptor}`

Top-level prefixes:

- `widget.*` -- embeddable widget strings
- `portal.*` -- public portal strings
- `common.*` -- shared between widget and portal (buttons, actions)

Examples:

```json
{
  "widget.shell.tab.feedback": "Feedback",
  "widget.shell.tab.changelog": "Changelog",
  "widget.shell.tab.help": "Help",
  "widget.shell.heading.feedback": "Share your ideas",
  "widget.shell.heading.help": "Help Center",
  "widget.shell.heading.changelog": "What's new",
  "widget.home.placeholder.title": "What's your idea?",
  "widget.home.placeholder.details": "Add more details...",
  "widget.home.emptyState.title": "No ideas yet",
  "widget.home.emptyState.description": "Be the first to share one!",
  "widget.home.search.placeholder": "Search ideas...",
  "widget.home.section.popular": "Popular ideas",
  "widget.home.section.similar": "Similar ideas",
  "widget.home.identity.postingAs": "Posting as {name}",
  "widget.home.identity.anonymous": "Posting anonymously",
  "portal.nav.feedback": "Feedback",
  "portal.nav.roadmap": "Roadmap",
  "portal.nav.changelog": "Changelog",
```

ICU MessageFormat syntax for interpolation (`{name}`), plurals (`{count, plural, one {# vote} other {# votes}}`), etc.

## IntlProvider Setup

Two independent `IntlProvider` mount points, one per surface (separate React trees):

### Portal

Mounted in `__root.tsx` or a portal layout wrapper:

- **Server-side:** read `Accept-Language` header in the TanStack Start loader, resolve to best available locale
- **Client-side:** hydrate with server-resolved locale. The `localStorage` override path is wired up for forward compatibility (future language switcher) but has no setter in this pass
- Wrap portal routes in `<IntlProvider locale={locale} messages={messages}>`

### Widget

Mounted in `widget-auth-provider.tsx`:

- **Primary:** accept `locale` from host app via `quackback:locale` postMessage or `?locale=` URL param
- **Fallback:** `navigator.language` in the iframe
- Wrap widget tree in its own `<IntlProvider>`

### Locale Resolution Order

| Surface | Priority 1                          | Priority 2           | Priority 3   |
| ------- | ----------------------------------- | -------------------- | ------------ |
| Portal  | `localStorage` override (future)    | `Accept-Language`    | `en` default |
| Widget  | SDK `locale` prop (postMessage/URL) | `navigator.language` | `en` default |

The `localStorage` slot is wired up in the resolution logic for forward compatibility, but without a language switcher UI in this pass, the effective portal resolution is `Accept-Language` then `en`.

### Message Loading

- English is bundled (always available, zero latency)
- Other locales are lazy-loaded via dynamic `import()` (e.g. `import(`../locales/${locale}.json`)`)
- If a requested locale file doesn't exist, fall back to `en`

### No Language Switcher UI

Locale is auto-detected or SDK-provided in this pass. A manual switcher can be added later.

## Component Migration Pattern

### JSX Strings

Before:

```tsx
{
  activeTab === 'feedback'
    ? 'Share your ideas'
    : activeTab === 'help'
      ? 'Help Center'
      : "What's new"
}
```

After:

```tsx
import { FormattedMessage } from 'react-intl'

{
  activeTab === 'feedback' ? (
    <FormattedMessage id="widget.shell.heading.feedback" defaultMessage="Share your ideas" />
  ) : activeTab === 'help' ? (
    <FormattedMessage id="widget.shell.heading.help" defaultMessage="Help Center" />
  ) : (
    <FormattedMessage id="widget.shell.heading.changelog" defaultMessage="What's new" />
  )
}
```

### Attributes (Placeholders, Aria Labels)

Use `useIntl()` hook for string contexts where JSX elements don't work:

```tsx
const intl = useIntl()

<input
  placeholder={intl.formatMessage({
    id: 'widget.home.placeholder.title',
    defaultMessage: "What's your idea?",
  })}
/>
```

### SEO Meta Tags

Portal pages have `<title>` and `<meta name="description">` tags. These must be translated using `intl.formatMessage()` in the route loaders/head config so search engines index the correct language.

### Date and Number Formatting

- Replace hardcoded `Intl.DateTimeFormat('en-US', ...)` with `intl.formatDate()` / `intl.formatNumber()` which use the active locale automatically
- `date-fns` usage in `time-ago.tsx` can stay as-is for now

### Migration Scope

**Migrated (widget + portal):**

- `apps/web/src/components/widget/*.tsx` (~8 files)
- Portal route pages and their shared components (~10-15 files)
- Shared UI components used by portal (status badges, vote buttons) -- strings only

**Not migrated:**

- Admin dashboard components
- Server-only code (API error messages, email templates)
- Developer-facing strings (console logs, error boundaries)

## SDK / Widget Locale Prop

### New PostMessage Type

Added to `WidgetInboundMessages` in `apps/web/src/lib/shared/widget/types.ts`:

```tsx
'quackback:locale': { locale: string }  // e.g. 'fr', 'ar', 'en-GB'
```

### Host App Usage

```js
widget.postMessage({ type: 'quackback:locale', locale: 'fr' }, '*')
```

### URL Param Alternative

The widget iframe URL accepts `?locale=fr` for simpler integrations. PostMessage overrides URL param if both are provided.

### Locale Handling in Widget

- Listen for `quackback:locale` in `widget-auth-provider.tsx` alongside existing message handlers
- Store locale in widget state
- On locale change, update `IntlProvider` props to trigger re-render
- Locale can be sent before or after widget opens

### Validation

- Normalize locale tags (e.g. `fr-FR` to `fr` if only `fr.json` exists)
- Unknown locales fall back to `en`

## RTL Support

Structural RTL only -- lay the groundwork so RTL locales work at a layout level without a dedicated audit of every directional edge case. Keyword data shows near-zero search volume from Arabic-speaking markets today, so full RTL polish is deferred until demand materializes.

### Dynamic Direction

Set `dir` and `lang` attributes dynamically on `<html>` (portal) and widget root:

```tsx
const rtlLocales = new Set(['ar', 'he', 'fa', 'ur'])
const dir = rtlLocales.has(locale) ? 'rtl' : 'ltr'

<html lang={locale} dir={dir}>
```

### Tailwind Logical Properties

Convert directional utilities to logical equivalents in migrated components during the string migration pass (nearly zero incremental cost):

| Physical                      | Logical                       |
| ----------------------------- | ----------------------------- |
| `ml-*` / `mr-*`               | `ms-*` / `me-*`               |
| `pl-*` / `pr-*`               | `ps-*` / `pe-*`               |
| `left-*` / `right-*`          | `start-*` / `end-*`           |
| `text-left` / `text-right`    | `text-start` / `text-end`     |
| `rounded-l-*` / `rounded-r-*` | `rounded-s-*` / `rounded-e-*` |
| `border-l-*` / `border-r-*`   | `border-s-*` / `border-e-*`   |

### RTL Debug Mode

Add `?rtl=1` debug query param that forces `dir="rtl"` with English strings for visual verification.

### Deferred to Follow-Up

The following are not included in this pass:

- Directional icon mirroring (chevrons, arrows)
- Scroll/gradient direction flipping
- Bidirectional text isolation (`<bdi>` wrapping)
- Full layout audit (absolute positioning, animation directions)

## Locale Persistence

Store the detected/resolved locale on the user's session (or anonymous session) so downstream systems (e.g. email notifications in a future pass) can reference the user's preferred language without re-detecting.

This is a lightweight addition: persist the resolved locale string alongside the existing session data. No new tables or APIs required.

## Translation Contribution Model

- Ship with English as the source of truth
- Community submits PRs with new locale JSON files
- Self-hosted operators can add/customize locale files in their deployment
- FormatJS CLI `intl:extract` generates a reference `en.json` that translators copy and translate
- An admin translation UI is a potential follow-up, not part of this spec

### Initial Locale Priority

Based on Ahrefs keyword data (native-language search volume for feedback tool keywords, difficulty scores):

1. **DE (German)** -- 450/mo "kundenfeedback", 150/mo "kundenfeedback software", difficulty 0-2
2. **PT-BR (Brazilian Portuguese)** -- 250/mo "feedback cliente", 200/mo "feedback do cliente", difficulty 0-1
3. **FR (French)** -- 40/mo "outil de feedback", plus 20/mo English-term searches from FR
4. **ES (Spanish)** -- 20/mo from ES/MX markets
5. **IT (Italian)** -- 20/mo from IT market

These are community-contributed, not blocking the implementation. The priority list informs which locale packs to solicit first.

## Out of Scope

- Admin dashboard i18n
- Content i18n (translating admin-authored data: board names, status labels, posts, help articles)
- Language switcher UI (relying on auto-detection from `Accept-Language` / SDK locale prop)
- Locale-prefixed URLs (`/fr/board/ideas`) for multilingual SEO (no competitors do this either)
- Email template translations (locale is persisted on session to enable this later)
- Full `date-fns` locale migration (relative time)
- Full RTL polish (icon mirroring, gradient flipping, bidi isolation, layout audit)
